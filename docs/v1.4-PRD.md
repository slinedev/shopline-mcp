## 1. 执行摘要

Shopline MCP Server（`shopline-mcp`）是一个 TypeScript/Node.js 实现的 MCP 服务器，通过 stdio 协议为 AI 客户端（如 Claude、Cursor 等）提供对 Shopline 电商平台 Open API 的直接访问能力。v1.3 兼容基线包含 **143 个业务工具**（75 读 + 68 写）和 **10 个 AI 助手工具**，覆盖 135 个 endpoint key。

v1.4 评审后修正目标为：**在不破坏 v1.3 兼容基线的前提下，追加商家运营核心覆盖**，包括 Webhooks、扩展 Settings、Live/Sales、User Coupons、Wish Lists、Staff、Return/Purchase 补全与选定 Metafields。官方 reference 当前约 315 个 method/path，因此本版不再承诺全站 95% 覆盖，也不纳入全量 Cart、Storefront/OAuth、Cart item metafields。

本 PRD 基于对 Shopline Open API 文档站点（v2.334.0）的索引和对现有代码库的分析。实施后项目提供 **269 个业务工具**（124 读 + 145 写）和 **10 个 AI 助手工具**，本地 endpoint table 映射 206 个 endpoint key，工具 metadata 覆盖 263 个 documented method/path。

## 3. 差距分析

### 3.1 完全缺失的模块

以下模块在文档中已有完整 API 参考，但项目中**零覆盖**：

| 模块 | 端点数量 | 业务价值 | 优先级 |
|------|---------|---------|--------|
| **Webhooks** | 5 (CRUD) | 实时事件集成，是 MCP 与外部系统桥接的关键 | 🔴 P0 |
| **设置 (Settings)** | ~20 | 店铺配置只读/写入，诊断和配置自动化必备 | 🔴 P0 |
| **直播/销售 (Live/Sales)** | 6 | 社交电商是 Shopline 亚洲市场核心差异化能力 | 🟡 P1 |
| **用户优惠券 (User Coupons)** | 5 | 客户级优惠券管理与核销 | 🟡 P1 |
| **心愿单 (Wish Lists)** | 3 | 客户行为数据，支持再营销 | 🟢 P2 |
| **购物车 (Carts)** | 5 | 购物车操作，支持 abandoned cart 恢复 | 🟢 P2 |
| **OAuth/Token 店面** | 8 | App 开发者场景 | 🟢 P3 |

### 3.2 部分覆盖的模块

| 模块 | 已覆盖 | 缺失 | 缺失内容 |
|------|--------|------|---------|
| **Metafields** | 1 | ~11 | Get/Update/Delete 多资源类型 + Definitions |
| **Return Orders** | 3 | 2 | Inspection execution + Delivery status update |
| **Purchase Orders** | 3 | 2 | Update PO + Create child PO |
| **Staff** | 1 | 2 | List all staff + Get staff detail |
| **Customers** | 13 | 2 | Coupon promotions list + Bulk store credit update |
| **Affiliate Campaigns** | 5 | 1 | Create campaign（端点已在 endpoints.ts 但缺工具注册）|

### 3.3 参数覆盖不足

部分现有工具未暴露文档中支持的完整参数集（如 `include_fields`、复杂筛选条件、特定物流选项等），需在 P1 阶段统一增强。

## 5. 功能拆解（按优先级）

### P0 — 核心覆盖补全

> **目标**: 将端点覆盖率从 ~75% 提升至 ~90%，补齐商家日常工作流的关键缺口。  
> **预计新增**: 30-50 个工具

#### 5.1 Webhooks 管理（5 端点 → 5 工具）

| 工具名 | 端点 | 模式 |
|--------|------|------|
| `list_webhooks` | `GET /v1/webhooks` | Read |
| `get_webhook` | `GET /v1/webhooks/{id}` | Read |
| `create_webhook` | `POST /v1/webhooks` | Write |
| `update_webhook` | `PUT /v1/webhooks/{id}` | Write |
| `delete_webhook` | `DELETE /v1/webhooks/{id}` | Write |

**关键设计决策**:
- Webhook 创建工具应提供 `topic` 枚举和 payload 示例说明（来自文档的 Webhook Topic and Payload Examples 页面）
- 应包含 webhook 最佳实践提示：签名验证、重试机制、IP 白名单
- 建议配套一个助手工作流 `webhook_setup`，引导商家完成 Webhook 的创建、验证、测试全流程

**需新增**:

| 工具名 | 资源类型 | 模式 |
|--------|---------|------|
| `get_merchant_app_metafields` | Merchant App | Read |
| `get_product_metafields` | Product | Read |
| `get_order_metafields` | Order | Read |
| `get_customer_metafields` | Customer | Read |
| `get_order_item_metafields` | Order Item | Read |
| `get_metafield_definitions` | Definitions | Read |
| `update_merchant_app_metafields` | Merchant App | Write |
| `update_product_metafields` | Product | Write |

### P2 — 体验、安全与多店深化

> **目标**: 不增加新 API 端点覆盖，深化现有功能的用户体验、安全控制和多店能力。  
> **预计新增**: 0-5 个工具（主要是增强）

#### 5.13 写操作安全增强

| 增强项 | 描述 |
|--------|------|
| **可配置审批强度** | 支持按工具类别配置：`SHOPLINE_APPROVAL_MODE=strict`（全量审批）/ `permissive`（仅高风险）/ `off`（开发环境）|
| **写操作影响预估** | 所有 Write 工具在 `dry_run` 时输出受影响资源数量（如"将修改 15 个 SKU 的库存"）|
| **批量操作限额** | 为批量写入添加硬上限保护（如单次最多更新 100 个商品），防止误操作 |
| **审批码 TTL** | `approval_code` 增加过期机制（默认 30 分钟），过期后必须重新生成 |

#### 5.14 错误处理增强

| 增强项 | 描述 |
|--------|------|
| **权限不足提示** | 当 API 返回 403 时，工具提示具体缺少哪个 Scope/Permission |
| **Rate Limit 感知** | 当收到 429 时，解析 `Retry-After` 头并提示等待时间，自动退避重试 |
| **参数校验前置** | 在发起 API 请求前对参数做 Zod 校验（如 email 格式、date 格式），减少无效请求 |
| **友好错误映射** | 将 Shopline API 常见错误码（`NOT_FOUND`、`UnprocessableEntityError` 等）映射为中文提示 |

#### 5.15 多店增强

| 增强项 | 描述 |
|--------|------|

| **store_alias 验证** | 启动时验证所有 store_alias 的 API Token 可用性，输出健康状态 |
| **统一审批队列** | 支持跨店审批：一个审批码可同时授权多个店铺的同一操作 |

## 8. 风险分析与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| **Shopline API 频率限制 (20 req/s)** | 高 | 中 | 现有 client.ts 已实现退避重试；新批量工具默认串行调用 |
